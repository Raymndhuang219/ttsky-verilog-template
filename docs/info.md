## How it works

This project is a reaction time tester implemented in Verilog. The design uses a single button to start and react, and displays the result in milliseconds on three multiplexed 7-segment displays.

The design cycles through four states:

- **IDLE** — The display shows `---`, indicating the device is ready. Press the button to start.
- **WAITING** — The display goes blank while the design waits for a random delay of 1–5 seconds, generated by a 16-bit LFSR. Pressing the button during this phase counts as an early press and resets back to IDLE.
- **GO** — The display shows `Go`, and the millisecond counter starts. Press the button as fast as possible.
- **RESULT** — The display shows your reaction time in milliseconds (0–999). Press the button again to reset and play another round.

The 8 output pins (`uo_out`) carry the 7-segment segment data. The three digit select pins (`uio_out[2:0]`) cycle through hundreds, tens, and ones units using multiplexing, which makes all three digits appear solid to the human eye simultaneously.

Timing is derived from the input clock frequency. The `CLK_HZ` parameter must match the actual board clock for accurate millisecond measurements. 

## How to test

1. Wire a button between `ui_in[0]` and VCC (active high). No external debounce is needed — the design includes a built-in synchroniser and rising-edge detector.
2. Wire three 7-segment displays:
   - Connect all segment pins (a–g, dp) in parallel across all three displays, driven by `uo_out[7:0]`.
   - Connect the cathode of the **hundreds** digit to `uio[2]`.
   - Connect the cathode of the **tens** digit to `uio[1]`.
   - Connect the cathode of the **units** digit to `uio[0]`.
3. Apply reset (`rst_n` low, then high).
4. The display should show `---`. Press the button to start.
5. Wait for the display to show `Go`, then press the button as fast as you can.
6. Your reaction time in milliseconds will appear on the display.
7. Press the button again to reset and play another round.

**Early press:** If you press the button before `Go` appears, the design resets silently back to `---`.

## Testbench

The testbench is written in Python using cocotb and consists of 5 self-checking tests. Each test prints a clear `PASS` or `FAIL` message with details. The simulation clock runs at 1 MHz (matching `CLK_HZ = 1_000_000` in the design) to keep simulation times practical while maintaining correct logical behavior.

The 5 tests and their justifications are:

**Test 1 — `test_idle_shows_dashes`:** Verifies that after reset, the design starts in IDLE with the dash pattern (`0x40`) on the segment output and all three digit select pins driven high. This confirms the reset logic initialises the state machine correctly and the display output is correct for the IDLE state.

**Test 2 — `test_early_press_resets`:** Presses the button to enter WAITING, then immediately presses again before GO appears. Verifies the display returns to dashes. This confirms the early press detection works correctly and the design safely resets to IDLE rather than getting stuck in an invalid state.

**Test 3 — `test_go_state_reached`:** After pressing start, polls the display for up to 6 seconds of simulation time and verifies that the GO state is eventually reached and the display shows the `G` or `O` segment pattern. This confirms the LFSR-based random delay and the second tick counter both work correctly.

**Test 4 — `test_reaction_time_displayed`:** Waits exactly 75ms after GO then presses the button and reads back the multiplexed 3-digit display. Verifies the displayed value is within ±2ms of 75. It confirms the millisecond counter increments accurately and the BCD conversion and 7-segment encoding are all correct.

**Test 5 — `test_result_press_returns_to_idle`:** After completing a full round (GO → react → RESULT), presses the button once more and verifies the display returns to dashes. This confirms the final state transition from RESULT back to IDLE works correctly, and that the design can be reused for multiple rounds without requiring a hard reset.

These tests check all four states of the design, all five possible state transitions, the timing accuracy of the millisecond counter, the 7-segment display encoding, the multiplexed digit select logic, and the LFSR random delay.
